# READMEレビューと改善アクションプラン（2026-03-20）

この文書は、以下READMEを読み込んだうえで、次にやるべき改善を優先度付きで整理した実行計画です。

- ../README.md
- ../application/README.md
- ../scoring-tool/README.md

## 1. いま分かっている現状

- 採点の土台は揃っている（ローカル計測手順、ビルド・起動手順あり）
- ただし、実運用で迷うポイントが README 間に分散している
- 現在のスコア課題は「性能指標そのもの」だけでなく「一部シナリオ未計測」が大きい
- 最近の改善で main.js は 2.02 MiB から 1.74 MiB まで削減済み（jqueryの初期読み込み外し）

## 2. README観点での改善ポイント（Pick up）

### A. 入口導線の改善（最優先）

問題:
- どの順番で何を実行すればよいかが、README -> docs/development -> scoring-tool README に分散

改善:
- ルート README に「最短5分クイックスタート」を追加
- 目的別導線（開発、計測、デプロイ、レギュレーション確認）を明示

期待効果:
- 初動ミスの削減
- チーム内で同じ手順に統一

### B. 計測オペレーションの標準化（最優先）

問題:
- 計測コマンドはあるが、失敗時の切り分け（未計測時の調査手順）がREADME上で弱い

改善:
- scoring-tool README に以下を追記
  - 代表的な targetName 一覧
  - 単体シナリオ計測の例
  - 未計測時の確認チェックリスト（認証、初期化API、セッション、起動URL）

期待効果:
- 「サインイン失敗」などの復旧時間短縮

### C. ワークスペース構成の表記ズレ修正（高優先）

問題:
- application README のディレクトリ構成説明が `/workspaces/...` になっており、実際の配置と認知ズレが起きやすい

改善:
- 実際のパス前提で表記を更新
- 主要ディレクトリ責務（client/server/e2e）を1行ずつ明記

期待効果:
- 新規参加者の誤読防止

### D. 競技向け運用ドキュメントの分離（高優先）

問題:
- 一般開発情報と競技運用情報（スコア差分管理、回帰防止）が混在

改善:
- docs に「performance-runbook」を新設
- 1改善1計測1記録のテンプレートを固定化

期待効果:
- 改善の再現性向上
- 回帰の早期発見

## 3. 技術面の次アクション（優先順位付き）

### P0: 未計測シナリオをゼロにする

対象:
- ユーザーフロー: ユーザー登録 -> サインアウト -> サインイン

理由:
- 採点仕様上、未計測は大きな失点
- 今の主要な取りこぼし要因

実施内容:
- scoring-tool の当該シナリオ期待動作と、認証モーダル挙動の差分確認
- 入力欄の自動補完やフォーム状態リセットの安定化
- 単体 targetName で再現 -> 修正 -> 再計測

### P1: ホームの表示スコア改善（FCP/SI/LCP/TBT）

対象:
- ホームを開く

理由:
- 現状のホーム得点が低く、総合得点の天井を下げている

実施内容:
- クリティカルレンダリングパスの削減
- さらに初期JSを分割（ホーム初期描画に不要な処理の遅延）
- レイアウトシフト対策（CLS安定化）

### P2: 操作シナリオの得点化（INP/TBT）

対象:
- 認証、DM、検索、Crok、投稿

理由:
- 250点満点の操作系スコアが大きい

実施内容:
- シナリオ別に blocking task を分解
- 1シナリオずつ操作遅延を削る

5. 結果を docs/development.md に記録する
